# Decisions

## 2026-05-05

### ログ設計の方針

現時点では、会話内容の全文はログに保存しない。

### 保存してよい情報

- 起動時刻
- 終了時刻
- API呼び出しの成功/失敗
- エラー種別
- 使用モデル名
- 会話回数

### 保存しない情報

- APIキー
- ユーザー入力の全文
- AI応答の全文
- 個人情報

### 理由

このプロジェクトはポートフォリオ用途の初期開発段階であり、個人情報や不要な会話内容を保存しない方が安全なため。

まずは、実行状況やエラー傾向を把握できる最小限のログにする。

### 最初のエージェント機能

最初のエージェント機能は、TODO整理エージェントにする。

### 目的

ユーザーの入力や現在の開発状況をもとに、次にやることを整理し、優先順位を提案できるようにする。

### 最初はやらないこと

- ファイルを自動編集する
- Git操作を自動実行する
- クラウド操作を行う
- 外部サービスと連携する

### 理由

危険な操作が少なく、現在のプロジェクト状況とも相性が良い。

まずは提案だけを行い、実行は人間が確認して行う。

## 2026-05-06

### ツール実行ルール

現時点では、AIエージェントはファイル編集・Git操作・クラウド操作などの実行系処理を自動では行わない。

### 許可すること

- プロジェクト情報の読み取り
- TODO整理の提案
- 次にやる作業の優先順位付け
- 実行手順の提案

### 自動実行しないこと

- ファイル編集
- Git操作
- クラウド操作
- 外部サービス連携
- 削除・停止・課金に関わる操作

### 実行前確認が必要なこと

将来的にツール実行を追加する場合、以下は必ずユーザー確認を挟む。

- ファイルを書き換える処理
- Git commit / push などの操作
- 外部APIを呼び出す処理
- クラウドリソースを作成・変更・削除する処理

### 理由

AIエージェントが危険な操作を自動実行しないようにするため。

まずは読み取りと提案に限定し、人間が内容を確認してから実行する方針にする。

### `src/` ディレクトリ分割方針

`main.py` に設定、OpenAI API呼び出し、プロジェクト文脈読み込み、エラー処理、CLIループなどの処理が集まってきたため、次の段階で `src/` ディレクトリへ分割する。

### 分割候補

- `src/config.py`: 設定値
- `src/openai_client.py`: OpenAI APIキー読み込み、クライアント作成、AI応答生成
- `src/context.py`: プロジェクト文脈ファイルの読み込み
- `src/agent.py`: TODO整理エージェントのプロンプト、初期会話履歴
- `src/stats.py`: 実行サマリー
- `src/errors.py`: エラー表示
- `src/cli.py`: CLIチャットループ

### 理由

今後、実行系ツールやログ機能を追加すると `main.py` がさらに大きくなるため、先に責務を分けて見通しを良くする。

### 方針

まずは動作を変えずにファイル分割だけを行う。
機能追加は分割後に行う。

## 2026-05-07

### 最初に追加する読み取り系ツール

最初に追加するツールは、プロジェクト状態サマリー表示ツールにする。

このツールは実行系ツールではなく、読み取り専用の補助ツールとして扱う。

### 目的

`TODO.md` / `docs/DEV_LOG.md` / `docs/DECISIONS.md` をもとに、現在の進捗と次にやる候補を整理して表示する。

### やること

- プロジェクト情報を読み取る
- 現在の進捗を要約する
- 次にやる候補を提案する

### やらないこと

- ファイル編集
- Git操作
- クラウド操作
- 外部API呼び出し
- 削除・停止・課金に関わる操作

### 理由

最初のツールとしては、読み取り専用で危険な操作が少ないため。
実行系ツールを追加する前に、まず安全な読み取り系ツールでツール設計に慣れる。


## 2026-05-09

### 最初の実行系ツールの検討方針

最初の実行系ツールは、危険な操作を避け、ローカル環境で完結する小さな処理から選ぶ。

### 候補

- TODO候補を表示するツール
- プロジェクト状態を整形して表示するツール
- ユーザー確認後にのみ実行する補助ツール

### 実行前確認

実行系ツールを追加する場合は、処理を実行する前に必ずユーザー確認を挟む。

### 禁止する操作

- ファイルを自動で書き換える処理
- Git commit / push などの操作
- クラウドリソースの作成・変更・削除
- 外部APIの呼び出し
- 削除・停止・課金に関わる操作

### 失敗時の表示

ツール実行に失敗した場合は、原因を短く日本語で表示する。
詳細なエラー情報や内部ログは、必要になった段階で開発者向けログとして分けて扱う。

### 方針

まずは読み取り系ツールの運用を安定させる。
実行系ツールは、安全性と確認フローを整理してから追加する。


### 実行系ツール候補の具体化

現時点では、以下の候補を検討対象にする。
いずれも、ファイル編集やGit操作は行わず、まずは提案・表示に限定する。

1. TODO候補生成ツール
   - 入力や現在のプロジェクト情報から、次に追加すべきTODO候補を表示する
   - ファイル編集はしない

2. コマンド候補表示ツール
   - 次に実行すべき確認コマンドを提案する
   - 実行はしない

3. ドキュメント更新案生成ツール
   - README / TODO / DEV_LOG に追記すべき文案を表示する
   - ファイル編集はしない

### 次に優先する候補

最初に具体化する候補は、TODO候補生成ツールにする。

理由は、現在のTODO整理エージェントと相性がよく、ファイル編集やGit操作を行わなくても価値を出せるため。

### TODO候補生成ツールの仕様

- 入力: 現在のプロジェクト情報
- 出力: 次に追加・確認すべきTODO候補
- 実行内容: 候補を表示するだけ
- やらないこと: TODO.mdの自動編集、Git操作、外部API呼び出し
- 実行前確認: ファイル編集をしないため不要
- 将来拡張: ユーザー確認後にTODO.mdへ追記できるようにするか検討
